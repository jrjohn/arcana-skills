# 設計審查指南

## 審查類型

| 類型 | 時機 | 參與者 | 時長 |
|------|------|--------|------|
| **自我審查** | 設計中 | 設計師本人 | 15-30 min |
| **同儕審查** | 初稿完成 | 2-3 位設計師 | 30-60 min |
| **跨職能審查** | 開發前 | 設計+開發+PM | 60 min |
| **QA 驗收** | 開發後 | 設計師 | 依規模 |

## Nielsen 10 啟發式

### 評估框架
| # | 原則 | 檢查點 |
|---|------|--------|
| 1 | **系統狀態可見** | 載入中、處理中、成功/失敗有回饋？ |
| 2 | **符合真實世界** | 用語使用者懂？比喻合理？ |
| 3 | **使用者控制** | 可取消？可復原？有緊急出口？ |
| 4 | **一致性與標準** | 相同功能相同設計？符合平台慣例？ |
| 5 | **預防錯誤** | 危險操作有確認？表單有驗證提示？ |
| 6 | **辨識而非記憶** | 選項可見？不需記住前頁資訊？ |
| 7 | **彈性與效率** | 有快捷操作？進階使用者能加速？ |
| 8 | **美學極簡** | 資訊精簡？無干擾元素？ |
| 9 | **錯誤處理** | 錯誤訊息清楚？提供解決方案？ |
| 10 | **幫助文件** | 需要時能找到說明？ |

### 嚴重度分級
| 等級 | 說明 | 處理 |
|------|------|------|
| 🔴 **Critical** | 無法完成任務 | 必須修復 |
| 🟠 **Major** | 嚴重影響體驗 | 應修復 |
| 🟡 **Minor** | 小困擾 | 排程修復 |
| 🟢 **Cosmetic** | 美觀建議 | 有空再改 |

## 設計 QA 檢查清單

### 視覺一致性
```
□ 顏色使用 Design Token
□ 字型符合規範
□ 間距符合 8px 柵格
□ 圖標風格一致
□ 圓角統一
□ 陰影層級正確
□ 響應式斷點正確
```

### 互動設計
```
□ 所有可點擊元素有 Hover 狀態
□ Focus 狀態清晰可見
□ 按鈕有 Loading 狀態
□ 表單有驗證回饋
□ 錯誤狀態已設計
□ 空狀態已設計
□ 骨架屏/載入狀態已設計
```

### 內容
```
□ 文案符合 UX Writing 規範
□ 沒有 Lorem ipsum
□ 考慮極端長度文字
□ 錯誤訊息有幫助
□ 標籤清楚明確
```

### 無障礙
```
□ 對比度 ≥ 4.5:1 (AA)
□ 觸控目標 ≥ 44px
□ 不只依賴顏色
□ 可鍵盤操作
□ 螢幕閱讀器友善
```

### 平台規範
```
iOS:
□ 符合 HIG
□ 正確的導航模式
□ SF Symbols 使用正確

Android:
□ 符合 Material Design 3
□ 正確的導航模式
□ Material Icons 使用正確

Web:
□ 響應式完整
□ 瀏覽器相容
□ 效能考量 (圖片大小)
```

## 審查流程

### 設計審查會議
```
準備 (會前):
├── 分享設計連結
├── 說明設計背景與目標
└── 列出希望獲得的回饋類型

進行 (會中):
├── 5 min: 設計師簡報背景
├── 15 min: 走過設計稿
├── 20 min: 收集回饋
└── 10 min: 整理行動項目

跟進 (會後):
├── 記錄所有回饋
├── 分類嚴重度
├── 安排修改時程
└── 更新設計稿
```

### 回饋格式
```markdown
## 回饋項目

### [嚴重度] 問題標題
**位置:** [畫面/元件名稱]
**問題:** [描述問題]
**影響:** [對使用者的影響]
**建議:** [可能的解決方案]

---
範例:
### 🟠 Major: 錯誤訊息不清楚
**位置:** 註冊表單 - Email 欄位
**問題:** 錯誤只顯示「無效」，沒說明原因
**影響:** 使用者不知道如何修正
**建議:** 改為「Email 格式錯誤，範例: name@example.com」
```

## 設計債務追蹤

### 什麼是設計債
```
├── 暫時性解法 (趕時間的妥協)
├── 不一致的設計 (舊元件未更新)
├── 缺少的狀態 (沒設計 Error 狀態)
├── 跳過的無障礙 (對比度不足)
└── 未實作的設計 (規格與實作有落差)
```

### 債務清單範本
```markdown
## 設計債務清單

| ID | 描述 | 嚴重度 | 影響範圍 | 預估工時 | 狀態 |
|----|------|--------|----------|----------|------|
| D-001 | Button 有 3 種不同圓角 | Minor | 全站 | 2h | Open |
| D-002 | 設定頁缺少 Dark Mode | Major | 設定 | 4h | In Progress |
| D-003 | 舊版 Modal 未更新 | Minor | 5 處 | 3h | Open |
```

### 償還策略
```
持續償還:
├── 每個 Sprint 分配 10-20% 償還設計債
├── 新功能同時更新相關舊元件
└── 定期設計系統維護 Sprint

優先級:
1. 影響使用者體驗的債務
2. 阻礙開發效率的債務
3. 美觀/一致性債務
```

```markdown
   - 缺點: ...

   - 優點: ...
   - 缺點: ...

## 決策
選擇 [選項 X]

[為什麼選這個？]

## 後果
[這個決策帶來什麼影響？]

- [設計稿連結]
- [相關 DDR]
```

## 驗收 Checklist

### 開發完成後驗收
```
視覺:
□ 顏色與設計稿一致
□ 字型大小/字重正確
□ 間距正確
□ 圖標正確
□ 響應式正確

互動:
□ 動畫/轉場正確
□ Hover/Focus 狀態正確
□ Loading 狀態正確
□ 錯誤狀態正確

功能:
□ 所有流程可走完
□ Edge case 處理
□ 效能可接受
```
